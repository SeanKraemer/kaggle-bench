# santander-customer-transaction-prediction Benchmark Notes

This task root defines the canonical action-bank benchmark task for the Santander Customer Transaction Prediction competition.

## Benchmark Shape

- Task artifacts live under `data/tasks/santander-customer-transaction-prediction/`.
- Machine-checked schemas live under `data/schema/`.
- Evaluator code and reports live under `eval/`.

## Task Contract

- This benchmark slice uses the official `train.csv` and `test.csv` files.
- The task-level bank lives in `candidate_actions.json`.
- Every testcase uses the same full bank.
- Each testcase stores only `input.context_action_ids`.
- The evaluator derives:
  - `candidate_action_ids = full_bank_action_ids - context_action_ids`
  - expected add units from the active good portion of the derived candidate side
  - expected remove units from the active bad portion of the context side

## Scoring Metadata

The bank may include hidden scoring metadata that should stay out of agent-facing benchmark renderings:

- `equivalence_group`
- `must_follow_action_ids`
- `invalidates_action_ids`
- `conflicts_with_action_ids`

These fields are for evaluator logic only.

## Scope

- `primary` scores only `eval_stage = core_preprocessing`
- `all` scores every active stage

## Primary Good steps

The three core preprocessing steps are:

1. **CA-000002** — Drop `ID_code` (string identifier with no predictive value)
2. **CA-000005** — Identify real vs synthetic test rows using per-feature value uniqueness
3. **CA-000009** — Create value-count features for each `var_*` from the combined train + real test distribution

CA-000005 must precede CA-000009.

## Key Competition Insight

Approximately half the 200,000 test rows are synthetically generated by the organisers. Synthetic rows can be identified because every feature value in such rows appears at least twice in the test set; real rows always have at least one feature value unique in the test set. Value-count features computed from train + real test dramatically improve AUC from ~0.90 to ~0.92+.

## Bank Snapshot

- Good actions: 3
- Bad actions: 11
- Primary effective good units: 3 (all at `core_preprocessing` stage)
- Bank satisfies `bad_count (11) >= 3 * primary_effective_good_unit_count (3)` gate

## Authoring Reminders

- All 200 `var_*` features are pre-normalized floats with no missing values; imputation and scaling steps are inapplicable.
- Shuffle augmentation (CA-000011) is a training-time technique, not a preprocessing step; it is excluded from the primary scope.
- PCA on the feature set (CA-000006) destroys the per-feature independence structure required for value-count features and it is a fault in this benchmark context.
