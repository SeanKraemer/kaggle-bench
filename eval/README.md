# Eval Benchmark

This directory is the home for the new action-bank benchmark evaluator.

The benchmark is defined by three layers:

- `data/tasks/`: benchmark tasks and authored artifacts
- `data/schema/`: machine-checked schemas for benchmark artifacts
- `eval/`: evaluator implementation, tests, and generated reports

## Benchmark Contract

The benchmark is a state-editing problem over a fixed task-level action bank.

Each testcase defines:

- `context_action_ids`
  - actions that are already present in the current state

The evaluator reads the full task-level bank from `candidate_actions.json` and derives:

- `full_bank_action_ids`
- `candidate_action_ids = full_bank_action_ids - context_action_ids`
- active scoring units under the selected stage scope
- expected add units from the active good portion of the bank
- expected remove units from the active bad portion of the context

Outputs predict:

- `predicted_add_action_ids`
- `predicted_remove_action_ids`

Output artifacts may also attach sidecars through `artifact_refs`.

- supported `kind` values are `trace`, `log`, `metadata`, `notebook`, `notes`, `prompt`, `stream`, and `other`
- use `prompt` for captured final prompt text sent to an agent or model
- use `stream` for raw streamed event logs such as jsonl session output
- the evaluator only reads `metadata` programmatically today; other sidecars are preserved for auditability and review

## Core Semantics

### Action Space

- every testcase uses the same full task-level bank
- testcase difficulty changes through `context_action_ids`
- add decisions are made from the derived candidate set
- remove decisions are made from the context set

### Source Of Truth

- testcase `context_action_ids` is the only testcase-level state input
- explicit `expected_add_action_ids` / `expected_remove_action_ids` are not stored
- scoring source of truth is derived from:
  - task-level candidate bank metadata
  - testcase `context_action_ids`
  - active stage scope

### Required Set Constraints

For every testcase:

- `context_action_ids ⊆ full_bank_action_ids`

For every output:

- `predicted_add_action_ids ⊆ full_bank_action_ids - context_action_ids`
- `predicted_remove_action_ids ⊆ context_action_ids`

Output lists are unordered unique sets:

- array order does not affect scoring
- duplicates should be rejected by schema validation
- if invalid duplicates still reach the evaluator, they should be counted as false positives

### Scope

The benchmark keeps the same two scope modes:

- `primary`
  - score only candidates with `eval_stage = core_preprocessing`
- `all`
  - score every active unit in the bank

When the evaluator is run in `primary` scope:

- derived expectations come only from `core_preprocessing` units
- predictions that only touch non-core units should be recorded as stage-filtered and ignored rather than counted as false positives

### Hidden Scoring Metadata

Some bank metadata is used by the evaluator but should not be surfaced to the agent as an explicit hint.

Examples:

- `equivalence_group`
- `must_follow_action_ids`
- `invalidates_action_ids`
- `conflicts_with_action_ids`

Agent-facing benchmark renderings should omit these hidden scoring fields.

### Candidate Metadata

Every candidate action should include:

- `action_id`
- `action_type`
- `canonical_params`
- `eval_stage`
- `role`
- `difficulty`
- `reasoning`
- `provenance_type`

Good candidates should also include:

- `notebook_refs`
- `source_profile`
- `context_notes`
- `evidence_snippets`
- `rareness`
- `confidence`

Other recommended optional fields:

- `base_refs`
- `derived_from_action_ids`
- `derivation_reasoning`
- `equivalence_group`
- `must_follow_action_ids`
- `invalidates_action_ids`
- `conflicts_with_action_ids`
- `challenge_type`
- `notes`

`role` values:

- `good`
- `bad`

`difficulty` values:

- `easy`
- `medium`
- `hard`

`provenance_type` values:

- `observed_notebook`
- `synthetic`

### Evidence Rules

- `good` candidates should use `observed_notebook` provenance
- `good` candidates should preserve strong supporting metadata comparable to curated golden actions
- `good` candidates should include direct `notebook_refs`, `source_profile`, `context_notes`, `evidence_snippets`, `rareness`, and `confidence`
- other `observed_notebook` candidates should include direct `notebook_refs`
- `synthetic` candidates may omit direct notebook refs, but must include traceable derivation evidence such as `base_refs`, `derived_from_action_ids`, and `derivation_reasoning`
- `synthetic` bad candidates are scored the same way as observed bad candidates

### Scoring Units

- by default, each candidate action is one scoring unit
- if multiple good candidates share an `equivalence_group`, they collapse into one add-side unit
- all members of an `equivalence_group` must share the same `eval_stage`
- any one valid member of an equivalence group can satisfy that unit
- predicting additional members from the same group should count as false positives

### Execution-Order Validity Rules

Outputs are unordered add/remove sets, so dependency and conflict checks are applied over the implied selected action set:

- `final_state_action_ids = (context_action_ids - predicted_remove_action_ids) ∪ predicted_add_action_ids`

The evaluator should construct a precedence graph over `final_state_action_ids` and ask whether at least one valid execution order exists.

`must_follow_action_ids` rule:

- if action `A` declares `must_follow_action_ids = [B, ...]`, every selected prerequisite should appear earlier than `A` in a valid execution order

`invalidates_action_ids` rule:

- if action `A` declares `invalidates_action_ids = [B, ...]` and both `A` and `B` are selected, then `B` must appear earlier than `A` in a valid execution order

`conflicts_with_action_ids` rule:

- if action `A` declares `conflicts_with_action_ids = [B, ...]` and both `A` and `B` are selected, then no valid execution order exists for that pair

Invalid predictions:

- should not satisfy recall
- should count as false positives

Role usage:

- `role` is used to derive target add/remove units
- `role` is not a standalone metric dimension in reporting

### Bank Composition Rules

- the bank should contain both `good` and `bad` actions
- the validator hard gate is `bad_count >= 3 * primary_effective_good_unit_count`
- `primary_effective_good_unit_count` means the number of `core_preprocessing` good units after collapsing `equivalence_group`s
- hard negative coverage matters; the bank should contain many hard bad actions
- near-miss and distractor style items should be concentrated in medium and hard difficulty bands

## Metrics

The evaluator should report:

- `add_precision`
- `add_recall`
- `add_f1`
- `remove_precision`
- `remove_recall`
- `remove_f1`

The evaluator should also expose the following headline summaries:

- `rollback_accuracy = remove_recall` (legacy alias retained for backward compatibility)
- `task_completion_score = 0.5 * add_f1 + 0.5 * remove_recall`

Recommended secondary metric:

- `strict_task_completion_score = 0.5 * add_f1 + 0.5 * remove_f1`

Binary success:

- use the same default threshold style as the existing benchmark
- `task_success = task_completion_score >= 0.5`

### Empty-Case Rules

Use the following zero-denominator convention:

- if the expected set is empty and the prediction is empty, score that side as `1.0`
- if the expected set is empty and the prediction is non-empty, precision is `0.0` and recall is `1.0` on that side

## Validation

Artifact validation should be implemented in:

- `eval/scripts/validate_artifacts.py`

This validator should check:

- schema conformance against `data/schema/`
- unknown action ids
- invalid add/remove membership
- invalid context membership
- invalid duplicate ids
- invalid dependency or conflict references
- same-stage `equivalence_group` membership
- bank-level valid execution order for representative good selections in `primary` and `all` scope
- selected-set valid execution order for testcase contexts and output final states
- candidate bank consistency
- artifact reference consistency

Evaluator behavior for invalid membership:

- invalid ids or invalid membership should not crash scoring
- they should be recorded and counted as false positives
